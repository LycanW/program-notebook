# 项目笔记本 (Program Notebook)

## 铁律

**不能凭记忆信任或更新 notebook。**

`docs/program-notebook.md` 和相关架构文档的内容必须来自当前代码、搜索结果、git 历史、构建配置、测试文件或已读项目文档。无法验证的内容只能标记为未知，不能补全猜测。

## 核心定位

`docs/program-notebook.md` 是项目中枢文档，负责记录全局地图、当前状态、关键决策、进度、风险、已知缺陷和参考资料索引。它不需要承载所有细节；当结构图、数据流、测试覆盖、协议说明或部署流程过长时，应拆分到 `docs/architecture/` 等专题文档，并在 notebook 中引用。

Notebook 的价值不是替代所有文档，而是让新成员和 AI 快速知道：项目现在是什么状态、重要资料在哪里、哪些事实已经验证、哪些风险需要注意。

## 文档分层

| 文档 | 职责 |
|------|------|
| `docs/program-notebook.md` | 中枢索引、项目摘要、当前状态、关键决策、进度、风险、已知缺陷、参考资料入口 |
| `docs/architecture/*.md` | 展开型专题文档：完整结构图、详细数据流、测试覆盖、CI、协议、驱动初始化、代码风格 |
| `docs/superpowers/specs/` | 设计规格、需求和技术决策过程 |
| `docs/superpowers/plans/` | 实现计划、任务拆解和执行记录 |
| `README.md` / `HOW2USE.md` | 面向人类使用者和开发者的入口说明 |
| `AGENTS.md` / `CLAUDE.md` | AI 代理需要遵守的项目约束、命令和注意事项 |

**规则：** Notebook 是目录和摘要，不是全部正文。专题文档更新后，notebook 必须同步索引和摘要；notebook 中引用的事实变化时，也必须检查对应专题文档是否需要更新。

## 适用与不适用场景

**适用：**

- 开始一个新项目，需要建立中枢文档
- 接手不熟悉的代码库，需要快速建立全局认知
- 项目文档分散、缺失或仅存在于口头
- 需要做跨模块改动，但缺乏全局视角
- 用户明确要求创建、更新或校验 program notebook
- 源码、配置、构建、测试、部署或协议发生结构性变化

**不适用：**

- 项目已有完备且维护良好的中枢文档和专题文档，且本次任务不涉及结构性变化
- 单文件或极简项目（少于 10 个源文件）
- 一次性脚本或临时工具
- 纯格式化、拼写或局部变量重命名，且不改变行为、接口或结构

## 强制工作流

1. 如果 `docs/program-notebook.md` 存在，跨模块改动前先读它。
2. 如果 notebook 不存在且项目非极简，先判断是否需要创建；不确定时询问用户。
3. 如果发现 `docs/architecture/`、`README.md`、`HOW2USE.md`、`AGENTS.md`、设计文档或计划文档，先把它们当作参考资料读取，不要重复造轮子。
4. 源码、配置、构建、测试或部署文件变更后，判断 notebook 和相关专题文档是否需要同步。
5. 最终回复或提交前，说明 notebook/docs 状态：已更新、无需更新、缺失但未创建，或仍有未验证项。

## 结构性改动触发条件

出现以下变化时，必须检查 notebook 和相关专题文档：

- 模块边界、入口点、公共 API、生命周期、任务/线程所有权变化
- 函数、服务、进程、设备之间的数据流变化
- 状态机、运行模式、错误处理、安全门、故障恢复变化
- 配置 schema、默认值、环境变量、命令行参数变化
- 构建脚本、代码生成、部署文件、CI 工作流变化
- 测试覆盖范围、测试架构、关键验证路径变化
- 已知限制、TODO/stub、验证过的缺陷被新增、修复或推翻
- 协议、驱动初始化、硬件交互、外部依赖的关键行为变化

通常不需要更新文档的情况：纯格式化或注释排版；局部变量重命名且不影响公开行为和调用链；只改测试断言文本；patch 级依赖升级且无行为、配置、构建影响；内部实现小优化且数据流、状态流、接口和风险不变。

## 何时创建或更新 `docs/architecture/`

当以下内容会让 notebook 过长或难以维护时，创建或更新 `docs/architecture/` 下的专题文档：

- 完整程序结构框图超过一个 Mermaid 图
- 数据流包含硬件、网络、异步任务、状态机、协议或多层反馈路径
- 测试覆盖需要按子系统、CI 阶段或验证链路说明
- CI/CD、部署、协议、驱动初始化、硬件故障排查需要独立维护
- 单个 notebook 章节超过约 100-150 行，影响 notebook 作为入口文档的可读性

推荐命名：

| 文件 | 用途 |
|------|------|
| `docs/architecture/01-program-structure.md` | 完整程序结构框图、模块职责、关键依赖 |
| `docs/architecture/02-data-flow.md` | 运行时数据流、硬件反馈、网络路径、状态传播 |
| `docs/architecture/03-ci-pipeline.md` | 测试覆盖、CI 验证链路、质量门 |
| `docs/architecture/04-code-style.md` | 代码风格、复杂模块、维护风险、重构建议 |

不要为了填表而创建空文档。只有当内容确实需要展开维护时才创建专题文档。

## Notebook 必需内容

Notebook 至少包含以下内容；细节过多时改为摘要 + 链接到专题文档。

### 1. 全局数据流摘要

用 Mermaid `graph LR` 展示核心数据流。颗粒度以关键模块、关键函数或关键任务为节点，连线说明数据、命令、状态或反馈的传递方向。只画项目自身的核心调用链，不追踪外部库内部细节。详细数据流超过一个图时，放到 `docs/architecture/02-data-flow.md`。

### 2. 架构摘要

用 Mermaid `graph TD` 或 `graph LR` 展示关键模块关系。每个节点必须至少在一条有效连线上；无关联的孤立节点不应出现在图中。

不展开结构体内部字段。字段名和类型代码里已有，notebook 复述只会过时。需要说明关键结构体时，用自然段描述设计要点、生命周期、所有权和边界情况。

完整结构框图、模块职责表和代码风格分析应放到 `docs/architecture/01-program-structure.md` 或 `04-code-style.md`。

### 3. 程序运行流摘要

用 Mermaid `sequenceDiagram` 或简短列表展示从入口函数到退出的生命周期：启动阶段、运行阶段、关闭阶段。

### 4. 配置加载链

记录配置来源、优先级、合并规则、覆盖策略和消费模块。关键回答两个问题：从哪里加载、谁用到它。

### 5. 逐模块摘要

按模块/目录分组，核心详写、辅助一行带过。大型项目用分组代替逐文件。

每个核心模块优先记录：入口点与调用链、设计决策和取舍、模块间关系和数据流向、陷阱和边界情况。

### 6. 参考资料索引

记录帮助理解项目的关键文件，不复制全文：架构专题、依赖声明、构建脚本、运行时配置、容器化和部署、AI 指南、使用说明、设计和计划。

### 7. 已验证的实现缺陷与限制

每一条都必须用代码证据验证。典型类型：死代码、未完成功能、架构漂移、文档漂移。

## 插件辅助检查

如果存在 `check_notebook` 工具，必须优先使用它获得文档状态报告：接手陌生项目时、跨模块改动前、结构性代码/配置/构建/测试变更后、最终声称 notebook 或 architecture docs 已同步前。

工具报告是证据入口，不是替代阅读。收到报告后仍需读取相关源码、配置或文档确认关键事实。

## 验证规则

**不推测。** 记录任何架构事实、缺陷、限制或风险前，必须用代码搜索、文件读取或 git 历史证明。

优先使用当前环境提供的搜索工具（`Grep`、`Glob` 或等价工具）。Shell 命令只是示例，不是强制要求。

无法验证的内容不写入，或明确标记为“未验证”。

## Mermaid 兼容性

| 用 | 不用 | 原因 |
|----|------|------|
| `graph LR` / `graph TD` | `flowchart` | 部分解析器较旧，不支持 v10+ |
| `stateDiagram-v2` | `stateDiagram` | v2 为现行标准 |
| 独立箭头 | `A --> B & C` | 多目标语法并非所有解析器支持 |
| ASCII 节点 ID | 中文节点 ID | Unicode ID 会导致部分解析器出错 |
| 虚线 `-.->` 表示实现关系 | | 比实线箭头更能表达 trait impl |

## 更新纪律

每次结构性改动后，对照本次改动逐项检查 notebook 和专题文档是否需要更新。需要更新的内容应尽量与代码在同一变更集中完成。

检查清单：

1. 新增/删除/移动模块或文件 → 更新 notebook 模块摘要；必要时更新 `01-program-structure.md`
2. 数据流、状态机、协议、驱动、异步任务变化 → 更新 notebook 摘要；必要时更新 `02-data-flow.md`
3. 新增配置、命令、部署步骤、构建行为 → 更新 notebook 配置链或参考索引
4. 测试、CI、质量门变化 → 更新 notebook 摘要；必要时更新 `03-ci-pipeline.md`
5. 重构风险、复杂模块、风格约定变化 → 更新 notebook 风险；必要时更新 `04-code-style.md`
6. 缺陷修复 → 从实现缺陷列表移除，或标记为已解决并附证据

## 文档陈旧检测

定期（或在跨模块改动前）检查：源文件目录与 notebook/architecture 模块列表是否一致；实现缺陷列表是否仍成立；`git log docs/program-notebook.md docs/architecture` 是否明显早于结构性源码改动；notebook 是否引用不存在或过期的专题文档。

## 输出报告格式

更新或校验后，用简短格式报告：

- Notebook 状态：已创建 / 已更新 / 已确认当前 / 已跳过（说明理由）
- 专题文档状态：已更新哪些 `docs/architecture/*.md`，或无需更新
- 检查证据：读过的文件、搜索结果、git log、`check_notebook` 报告
- 变更章节：数据流 / 架构 / 运行流 / 配置 / 模块 / 参考资料 / 缺陷
- 未验证项：只在确实无法验证时列出

## 红灯信号——停下来验证

| 借口 | 事实 |
|------|------|
| “这个改动太小，不更新也行” | 小改动累积 = 文档慢性死亡。先判断是否结构性变化 |
| “稍后一起更新” | 稍后 = 永不。上下文会丢失，细节会遗忘 |
| “只是重构，结构没变” | 重构经常改变模块边界、调用链和风险 |
| “文件太多，写不全” | 按模块分组即可，不必逐文件。必要时拆专题文档 |
| “我已经在脑子里更新了” | 文档价值在于别人和未来的 AI 能读到 |
| “notebook 已经很长了” | 长不是理由；应该拆到 `docs/architecture/` 并建立索引 |
| “插件提醒了，但这次可以忽略” | 提醒意味着需要判断；忽略前必须说明无需更新的证据 |

**违反更新纪律的字面规则就是违反其精神。**

## 常见错误

- 推测性缺陷：没有用代码搜索证明就写“某功能缺失”
- 箭头方向反了：实现指向接口（trait/interface），不是反过来
- 子图与节点 ID 冲突：`subgraph CONFIG[...]` 再加节点 `CONFIG` 会导致 Mermaid 循环错误
- 重构后未更新：架构变动后 notebook 或专题文档不同步
- 遗漏参考文件：配置 schema、构建脚本、CI 和 AGENTS 指南的重要性不亚于源码
- 大型项目逐文件展开：500 个文件不需要 500 条摘要，按模块分组
- 结构体字段展开：字段名和类型代码里已有，用自然段描述设计要点和分组
- 孤立节点图：图里的每个节点必须至少在一条线上
- 把专题文档当作 notebook 替代品：专题文档必须被 notebook 索引和摘要
